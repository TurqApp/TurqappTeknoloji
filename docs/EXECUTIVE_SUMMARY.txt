5. CRITICAL SECURITY ISSUES
================================================================================

CRITICAL (Do Not Ship):
  1. Plaintext Password Storage in Firestore
     - Field: "sifre" in users collection
     - Risk: Credential exposure in database console
     - Impact: CRITICAL - User accounts at risk
     - Solution: Use Firebase Auth, remove plaintext

  2. Sensitive Data Without Encryption
     - Family information (parent names, jobs, salaries)
     - Financial data (IBAN, bank accounts)
     - Medical information (disability reports)
     - Housing type (for scholarship determination)
     - Solution: Encrypt sensitive fields at rest

  3. No Rate Limiting
     - Login attempts: No visible brute-force protection
     - API calls: Users could spam posts, comments
     - Solution: Implement rate limiting (client + server)

HIGH PRIORITY:
  4. Large Unbounded Arrays in User Document
     - blockedUsers[], readStories[], lastSearchList[]
     - Can grow to 100KB+ after 10k items
     - Solution: Move to sub-collections

  5. Listener Leak Risk (315 listeners)
     - Not all unsubscribed in onClose()
     - Memory leaks possible
     - Wasted read quota

================================================================================
6. PERFORMANCE BOTTLENECKS
================================================================================

CRITICAL IMPACT:
  1. N+1 Query Problem (10-50x amplification)
     - Fetching feed: load post → fetch user → fetch followers
     - Impact: Massive read quota waste
     - Solution: Cache layer or denormalize user data in posts

  2. Missing Composite Indexes (MAJOR)
     - Complex queries on (timeStamp, userID, privacy) slow
     - Need explicit index strategy
     - Solution: Add Firestore indexes

  3. Large User Document (130+ fields)
     - Every user read = ~25KB minimum
     - Impact: Slow load times, high bandwidth
     - Solution: Normalize into sub-collections

MODERATE IMPACT:
  4. Redundant Collections
     - liked_posts stored in 2 places
     - Consistency risk, write duplication
     - Solution: Single source of truth

  5. No Query Batching
     - Individual reads instead of Future.wait()
     - Higher latency
     - Solution: Batch multiple queries

  6. Real-time Listeners on Same Paths
     - Multiple listeners watching same collections
     - Unnecessary reads
     - Solution: Consolidate listeners

OPTIMIZATION OPPORTUNITIES:
  7. 7-day Cache TTL for User Data
     - Data potentially stale for week
     - Solution: Shorter TTL or event-based invalidation

  8. Soft Delete Strategy
     - Every query needs "deletedPost != true" filter
     - Solution: Separate collection for deleted items

10. SCALABILITY ASSESSMENT
================================================================================

Current Capacity Estimates (at 100K users):
  User Collection:       2-5 GB
  Posts Collection:      10-50 GB
  Interaction Data:      5-10 GB
  Total Estimated:       50-200 GB

Read Operations Per Day:
  Estimated:            5-50 million reads
  At current patterns:  10-50x amplified due to N+1

Firestore Quota Impact:
  Cost:                 ~$200-2000/month at high scale
  Write Operations:     100K+ per day
  Storage:              Approaching limits at scale

Recommendations:
  1. Implement caching layer (Redis/Memcache)
  2. Use Cloud Functions for aggregations
  3. Move analytics to BigQuery
  4. Implement sharding for popular documents
  5. Archive old posts (>1 year)
  6. Connection pooling for database
